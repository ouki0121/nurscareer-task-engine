CAのためのアセット経営ガイド
新規リードだけに頼らない。自分のアセットを積み上げて、安定した売上基盤を作る。

第1部：前提知識 — まず言葉を揃える

なぜアセットマネジメントが重要なのか？

セールスには2つの思考法がある。PL脳（損益計算書的思考）とBS脳（貸借対照表的思考）。
どちらが正解ではなく、PL脳だけだと限界が来る。BS脳を持つことで売上が安定する。

各取引を「木」に例えるとわかりやすい
🍎
PL脳
損益計算書的思考
視点 実った果実を摘むだけ
行動 即収穫・即離脱
結果 一時的な収益
リスク 土壌劣化（顧客離反）
「今月の新規リードから何件決まるか」
だけを考えている状態
🌳
BS脳
貸借対照表的思考
視点 苗木を植えて育てる
行動 水やり・肥料（フォロー）・剪定（関係深化）
結果 毎年実る果実 + 新しい枝（クロスセル・紹介）
リスク 初期投資が必要だが長期安定
「自分のアセットをどう積み上げるか」
を考えている状態
PL脳だけでは、毎月ゼロからのスタート。
BS脳を持てば、過去の積み上げが今月の売上を作る。
SQLアセット（蛇口B）と成約者アセット（蛇口D）を育てることが、BS脳の実践。
このアセットを管理するためのフレームワークが「ヨミ表（アセット）マネジメント」。

これは「求職者管理」ではありません。「取引管理」です。

ヨミ表の1行は「1人の求職者」ではなく 「1つの取引」 です。
同じ求職者でも、転職活動が別なら別の取引（別の行）になります。
だから「掘り起こし」や「再転職」のときは、既存の行を書き換えずに新しい行を追加します。

Deal

取引が受注に至るまでのベルトコンベア。1つだけ。全取引がここを通る。
蛇口
Source
パイプラインに取引を投入する4つの入口。今月どの蛇口を開けるかを考えるのがアセット経営。
アセット
Asset
CAが経験と共に積み上げる資産。SQLアセット（時期先の取引）と成約者アセット（過去の成約者との関係性）。ランクを上げる鍵。
MQLプール
MQL Pool
担当なしの共有案件。誰でも掘り起こしOK。早い者勝ち。止まった取引は全てここに戻る。

そのステータスから受注に至る確率。求まる=33% / 面接セット=35% / 内定提示=90%。
ステータス

不良在庫

動いていない取引。放置するとパイプラインが詰まり、新しい案件に手が回らなくなる。

理想のセールスサイクル

フォーム回答から内定承諾まで、理想は30〜45日。
求まるから面接セットは7日以内。面接後フォローは当日中。
内定が出たら即日クロージング。スピードが受注率を決める。

3つの循環ルート

1. MQLルート: 止まった取引 → MQLプール → 蛇口Cとして再供給
2. 成約者ルート: 入金 → フォロー → 成約者アセットに蓄積 → 蛇口D
3. リファラルルート: 成約者に紹介を依頼 → 新規リードとして蛇口Aに
この循環が回り続ける限り、パイプラインは枯れない。

フォーム回答 → 求まる: 7〜14日
求まる → 面接セット: 7日以内
面接 → 内定提示: 3〜7日
内定提示 → 承諾: 1〜3日
承諾 → 入金: 60〜90日
入金後 → フォロー: 2ヶ月スパン

1つのリードの一生 — 全パターンのフローチャート

広告から1人のリードが入ったとき、どんな道をたどる可能性があるか。
全ての分岐を知っておくことで、「次に何をすべきか」が見える。

アセットを積み上げる = キャリアランクを上げる

新規リード（蛇口A）とMQLプール（蛇口C）は、今月の売上を作る「フロー型」の蛇口。
SQLアセット（蛇口B）と成約者アセット（蛇口D）は、経験と共に積み上がる「ストック型」の蛇口。
このストック型アセットを厚くすることが、新規リードに依存しない安定した売上基盤を作り、キャリアランクを上げる長期戦略。

SQLアセットを積み上げるには

過去に割り振られたリード1人1人を大切に、丁寧に管理する。
「今は時期じゃない」を捨てずにSQLとしてホールド。受注月を明記。
時期が来たら確実にアタック。ナーチャリングの積み重ね＝SQLアセットの厚み。

成約者アセットを積み上げるには

入金後の返金規定終了後フォロー、関係性のキープが全て。
この成約者アセットはリファラルの供給源にもなるし、再転職としてダイレクトの受注にもつながる。
フォローは営業コストではなく、将来の売上への投資。

第3部：運用ルール — ヨミ表を正しく管理する

ステータス変更の判断基準

フォーム回答→SQL転職時期が先だが自分の取引として確保。受注月を必ず明記（SQLアセットになる）
フォーム回答→MQL転職時期不明・対応不要。MQLプールに戻して誰でも掘り起こし可能に
フォーム回答→求まる済み初回アポ完了 + 求人探索完了 + 求人紹介済み
SQL（時期到来）→求まる済み受注月が到来。アタック → アポ → 求人紹介 → 求まる取得
求まる済み→面接セット済み事まる取得 + 面接日程が確定
面接セット済み→内定提示面接合格 + 事業者から内定が出た
面接セット済み→不合格・再調整中面接不合格。別求人を探索 or MQLリリース
内定提示→内定承諾求職者が内定を承諾した
内定承諾→請求管理シート反映済み★請求管理シートに請求情報を記載完了
請求管理シート反映済み→入金確認済み入金を確認。入社後フォロー開始（成約者アセットに蓄積）
どこからでも→MQL対応不要・紹介不可・音信不通・失注 → すべてMQLプールに戻す（クローズはない）

第4部：AIがサポートする — だからヨミ表を正しく管理する

毎朝8時、Slackにタスクが届く

AIがヨミ表を読み取り、あなた専用のタスクリストを自動生成します。
棚卸し・優先順位付け・異常検知・目標ギャップ計算 — 全部自動。
あなたは「考える」時間をゼロにして「動く」に集中できる。

4つの蛇口の棚卸し

各蛇口の取引数を自動カウント。目標との差分を毎朝表示

優先順位付け

内定提示 → 面接フォロー → 求まる推進 → ... の順でタスクを並べる

異常検知

フォーム10件超・求まる7日超・SQLアセット20件超を自動警告

ただし条件がある

ヨミ表のステータスが正しくないと、AIも正しく動けない。
アセットの鮮度管理はCAの責任。
