ヨミ表 棚卸しガイドライン

CA 1-on-1 棚卸しセッション用 ｜ ステータス定義・変更ルール・新レコード作成ルール

棚卸しの目的

ヨミ表の全レコードを見直し、ステータスを現実と一致させる。
正しいステータス = AIが正しいタスクを出せる = あなたの売上が上がる。

棚卸し前に確認: ヨミ表の1行 = 1つの「取引」。求職者1人ではない。
同じ人でも別の転職活動なら別の行。既存の行を書き換えるのではなく、新しい行を追加するケースがある（後述）。

ステータス定義（M列）— 全11種

「今この取引はどの状態か？」で選ぶ。迷ったら下の判断フローを参照。

フォーム回答
担当制
問い合わせが入り、まだ初回アポ〜求人紹介に至っていない状態。
やること: 即日架電。アポを取って求人探索へ進める
SQL
担当制（自分の資産）
転職時期が先だが、自分の取引として確保しておきたい案件。
G列（受注予測月）を必ず明記すること。受注月が来たらアタック開始。
やること: 受注月の1ヶ月前にアタック開始。それまでナーチャリング
求まる済み
担当制 / 受注角度33%
求人紹介が完了し、事業者から「この候補者で進めたい」と了承（求まる）を得た状態。
やること: 7日以内に事まる取得→面接日程確定を急ぐ
面接セット済み
担当制 / 受注角度35%
面接日程が確定している状態。X列（面接予定日）に日付を入れる。
やること: 面接当日中に結果確認→即フォロー→クロージング
内定提示
担当制 / 受注角度90%
事業者から内定が出ている状態。あとは求職者の承諾を取るだけ。
やること: 即日クロージング。承諾を取り切る
内定承諾
担当制 / 受注角度100%
求職者が内定を承諾した状態。請求管理シートへの移行が必要。
やること: ★請求管理シートに請求情報を記載→ステータスを「請求管理シート反映済み」に変更
不合格・再調整中
担当制
面接で不合格。別の求人を探索して再チャレンジ中。
やること: 別求人で求まる取得→「求まる済み」に復帰。紹介不可なら「MQL」にリリース
早期退職（返金）
担当制
入職後に早期退職・入職辞退が発生。返金処理中。
やること: 再転職意向を確認。意向あり→新レコード作成。なし→MQLリリース
請求管理シート反映済み
—
請求情報が★請求管理シートに記載済み。入金待ち。
やること: 入金確認したら「入金確認済み」に変更
入金確認済み
—
入金完了。入社後フォロー対象（成約者アセット）。
やること: 入職月+2ヶ月からフォロー開始。2ヶ月スパンで定期連絡
MQL
共有（担当なし）
担当がいない共有プール。誰でも掘り起こしOK。早い者勝ち。
音信不通・失注・対応不要・紹介不可 → すべてここに戻す。「クローズ」はない。
やること: 目標にギャップがあるとき、ここから掘り起こす

廃止ステータス — 見つけたら変更する

以下のステータスが残っていたら、棚卸しで正しいステータスに変更してください。

取引クローズ→MQLクローズという概念はない。全てMQLに戻す
失注（他社流れ）→MQL他社に決まっても将来掘り起こし可能
音信不通→MQL連絡が取れなくなった。MQLに戻す
対象外→MQLMQLに戻す
転職取りやめ→MQL将来また転職するかもしれない。MQLに
着席→状況によるフォーム回答 or 求まる済みに変更
受注済み顧客→内定承諾請求管理シートに移行してステータス変更
1/2/3回目フォロー済み→入金確認済みフォローはAIが管理。ステータスは入金確認済みに統一

棚卸し判断フロー — 迷ったらこれで判断

Q1. この取引、今動いてる？（直近30日以内にアクションした？）
はい → 現在のステータスが正しいか確認
U〜Z列の日付とM列のステータスが一致しているかチェック
いいえ → Q2へ
30日以上動いてない取引は整理が必要
Q2. この人、自分の取引として持っておきたい？
はい → SQLに変更
G列に受注予測月を必ず入れる。いつアタックするか明記する
いいえ → MQLに変更
MQLプールに戻す。誰かが掘り起こせるようにリリース
Q3. フォーム回答のまま10件以上残ってない？
10件以上ある → 整理が必要
1件ずつ: アポ済み→求まるに進める / 時期先→SQL / 不要→MQL
10件未満 → OK
正常範囲。フォーム回答は10件以下が目安
Q4. SQLが20件以上ない？受注予測月が過去になってるものない？
ある → 受注月を更新 or MQLに
一度連絡して状況確認。受注月を最新に更新。もう不要ならMQLに
ない → OK
SQL20件以下かつ受注月が最新ならOK

ステータス変更の早見表

フォーム回答→SQL転職時期が先。自分の取引として確保。受注月を必ず明記
フォーム回答→MQL対応不要・時期不明。プールにリリース
フォーム回答→求まる済み初回アポ済み + 求人紹介 + 求まる取得
SQL（時期到来）→求まる済み受注月到来。アタック→アポ→求人紹介→求まる
求まる済み→面接セット済み事まる取得 + 面接日程確定。X列に面接日を入力
面接セット済み→内定提示面接合格。事業者から内定が出た
面接セット済み→不合格・再調整中面接不合格。別求人探索 or MQLリリース
内定提示→内定承諾求職者が承諾した
内定承諾→請求管理シート反映済み★請求管理シートに記載完了
請求管理シート反映済み→入金確認済み入金を確認
どこからでも→MQL対応不要・紹介不可・音信不通・失注 → 全てMQLへ

新しい行を作るタイミング

既存の行は書き換えない。新しい行を追加する。元の行はそのまま残す。

① MQL掘り起こし

MQLステータスのものを掘り起こし、SQLとして確保する場合

1
MQLプールから候補を見つけ、コンタクトを取る
2
ヨミ表に新しい行を追加
3
I列（新規流入チャネル）= 「掘り起こし」
4
M列 = SQL（時期先の場合）or フォーム回答（すぐ動ける場合）
5
E列 = 自分の名前 / U列 = コンタクト取れた日 / G列 = 受注予測月
!
元のMQLレコードは触らない

② 再転職

以下のいずれかのステータスの人が再び転職する場合。全て同じ手順。

早期退職（返金） 請求管理シート反映済み 入金確認済み
1
再転職の意向を確認（フォロー or 本人からの連絡）
2
意向あり → ヨミ表に新しい行を追加
3
I列（新規流入チャネル）= 「再転職」
4
M列 = SQL / U列 = 再転職を進める言質を取れた日 / G列 = 受注予測月
5
元の取引レコードはそのまま残す
!
意向なし（早期退職の場合）→ MQLにリリース

③ リファラル（知人紹介）

どのステータスの成約者からもあり得る。紹介された知人とコンタクトが取れた時点で作成。

1
成約者から知人の紹介を受け、紹介者とコンタクトを取る
2
ヨミ表に新しい行を追加
3
I列（新規流入チャネル）= 「知人紹介」
4
U列 = 紹介者とコンタクトが取れた日
5
以降は通常の新規リードと同じフロー

1-on-1 棚卸しチェックリスト

CAと一緒に、上から順にチェックしていく。

  • 廃止ステータス（取引クローズ・失注・音信不通・転職取りやめ等）が残っていないか → MQLに変更
  • フォーム回答が10件以上残っていないか → SQL or MQLに整理
  • フォーム回答で30日以上経過しているものはないか → SQL or MQL
  • SQLの受注予測月（G列）は全件入っているか → 入っていないものは明記する
  • SQLの受注予測月が過去日付のものはないか → 更新 or MQL
  • SQLが20件を超えていないか → 超えていたら受注月の精度を見直す
  • 求まる済みで7日以上経過しているものはないか → 事まる取得を急ぐ
  • 内定承諾のまま請求管理シートに未移行のものはないか → 移行する
  • 入金確認済みで入社後フォローが必要な人はいないか → フォロー計画を確認
  • 早期退職（返金）の案件はどうする？ → 再転職 or MQLリリース
