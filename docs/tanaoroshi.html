ヨミ表 棚卸し — 初期セットアップ

CA 1-on-1 棚卸しセッション用 ｜ 今のデータを正しい状態にする一度きりの作業

棚卸しの目的

ヨミ表の全レコードを見直し、ステータスを現実と一致させる。
正しいステータス = AIが正しいタスクを出せる = あなたの売上が上がる。

棚卸し前に確認: ヨミ表の1行 = 1つの「取引」。求職者1人ではない。
同じ人でも別の転職活動なら別の行。既存の行を書き換えるのではなく、新しい行を追加するケースがある（後述）。

廃止ステータス — 見つけたら変更する

以下のステータスが残っていたら、棚卸しで正しいステータスに変更してください。

取引クローズ→MQLクローズという概念はない。全てMQLに戻す
失注（他社流れ）→MQL他社に決まっても将来掘り起こし可能
音信不通→MQL連絡が取れなくなった。MQLに戻す
対象外→MQLMQLに戻す
転職取りやめ→MQL将来また転職するかもしれない。MQLに
着席→状況によるフォーム回答 or 求まる済みに変更
受注済み顧客→内定承諾請求管理シートに移行してステータス変更
1/2/3回目フォロー済み→入金確認済みフォローはAIが管理。ステータスは入金確認済みに統一

棚卸し判断フロー — 迷ったらこれで判断

Q1. この取引、今動いてる？（直近30日以内にアクションした？）
はい → 現在のステータスが正しいか確認
U〜Z列の日付とM列のステータスが一致しているかチェック
いいえ → Q2へ
30日以上動いてない取引は整理が必要
Q2. この人、自分の取引として持っておきたい？
はい → SQLに変更
G列に受注予測月を必ず入れる。いつアタックするか明記する
いいえ → MQLに変更
MQLプールに戻す。誰かが掘り起こせるようにリリース
Q3. フォーム回答のまま10件以上残ってない？
10件以上ある → 整理が必要
1件ずつ: アポ済み→求まるに進める / 時期先→SQL / 不要→MQL
10件未満 → OK
正常範囲。フォーム回答は10件以下が目安
Q4. SQLが20件以上ない？受注予測月が過去になってるものない？
ある → 受注月を更新 or MQLに
一度連絡して状況確認。受注月を最新に更新。もう不要ならMQLに
ない → OK
SQL20件以下かつ受注月が最新ならOK

1-on-1 棚卸しチェックリスト

CAと一緒に、上から順にチェックしていく。

  • 廃止ステータス（取引クローズ・失注・音信不通・転職取りやめ等）が残っていないか → MQLに変更
  • フォーム回答が10件以上残っていないか → SQL or MQLに整理
  • フォーム回答で30日以上経過しているものはないか → SQL or MQL
  • SQLの受注予測月（G列）は全件入っているか → 入っていないものは明記する
  • SQLの受注予測月が過去日付のものはないか → 更新 or MQL
  • SQLが20件を超えていないか → 超えていたら受注月の精度を見直す
  • 求まる済みで7日以上経過しているものはないか → 事まる取得を急ぐ
  • 内定承諾のまま請求管理シートに未移行のものはないか → 移行する
  • 入金確認済みで入社後フォローが必要な人はいないか → フォロー計画を確認
  • 早期退職（返金）の案件はどうする？ → 再転職 or MQLリリース
棚卸しが終わったら → ヨミ表マネジメントガイドライン を日常運用の参考にしてください
